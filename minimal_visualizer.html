How Glider-Based Computation Works

1. The Ether Background

Rule 110 has a stable periodic background called ether: a 14-cell repeating pattern 00010011011111. When evolved, ether remains ether — it's the "vacuum" of Rule 110.

2. Encoding Inputs as Gliders

To encode a 1 bit, we inject a "phase-shifted bubble" into the ether at a specific position. This creates a glider — a localized disturbance that propagates through the ether over time.

To encode a 0 bit, we leave pure ether (no injection).

Input bit = 1 → inject bubble → glider propagates leftward
Input bit = 0 → pure ether → no propagation

3. Glider Collisions = Logic

When two gliders collide, they interact and produce new gliders (or annihilate). This is the basis of computation:

  • XOR-like behavior: Two gliders colliding may annihilate (1⊕1=0), while a single glider passes through (1⊕0=1)
  • AND-like behavior: Certain collision configurations only produce output if both inputs are present

By carefully positioning input gliders, we can arrange for their collisions to compute logic gates.

4. Reading the Output

After enough evolution steps, we examine specific output regions:

  • Disturbance present (differs from evolved pure ether) → output = 1
  • No disturbance (matches evolved pure ether) → output = 0

Compare final_state[region] vs pure_ether_evolved[region]
If different → bit = 1, If same → bit = 0

5. Why This Implementation is Incomplete

For addition, we need a full adder per bit with carry propagation:

Sum[i] = A[i] ⊕ B[i] ⊕ Carry[i-1]
Carry[i] = (A[i] ∧ B[i]) ∨ (Carry[i-1] ∧ (A[i] ⊕ B[i]))

The current layout injects gliders for A and B bits, but doesn't properly route carry signals between bit positions. The gliders collide chaotically rather than implementing the adder equations.

What's needed: Precise spatial layout where:

  • XOR collisions happen at sum output positions
  • AND collisions feed into carry chain
  • Carry gliders arrive at the next bit position at the right time
